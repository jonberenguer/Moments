## FFmpeg Setup

FFmpeg must be placed in `bin/` before building. It is bundled into the installer and shipped to users вЂ” they never need to install it separately.

### Recommended: download script (BtbN GPL builds)

```bash
node scripts/download-ffmpeg.js          # both platforms (default)
node scripts/download-ffmpeg.js linux    # Linux x64 only  в†’ bin/linux/ffmpeg
node scripts/download-ffmpeg.js win      # Windows x64 only в†’ bin/win/ffmpeg.exe
```

Downloads and unpacks the [BtbN GPL builds](https://github.com/BtbN/FFmpeg-Builds/releases). These include the **`drawtext`** filter (required for text overlays) plus `libx264` and the `h264_nvenc`/`h264_amf`/`h264_qsv` GPU encoders. Extraction needs `tar` (with xz) and `unzip` вЂ” on Debian/Ubuntu: `apt-get install -y xz-utils unzip`.

> вљ пёЏ Do **not** use the `ffmpeg-static` npm package. Its bundled (johnvansickle) static build omits the `drawtext` filter, so text-overlay export fails with `No such filter: 'drawtext'`.

### Manual alternative

**Linux:**
```bash
curl -L https://github.com/BtbN/FFmpeg-Builds/releases/download/latest/ffmpeg-master-latest-linux64-gpl.tar.xz \
  -o /tmp/ffmpeg-linux.tar.xz
tar -xf /tmp/ffmpeg-linux.tar.xz -C /tmp/
mkdir -p bin/linux
cp /tmp/ffmpeg-master-latest-linux64-gpl/bin/ffmpeg bin/linux/ffmpeg
chmod +x bin/linux/ffmpeg
```

**Windows (from your Linux container):**
```bash
curl -L https://github.com/BtbN/FFmpeg-Builds/releases/download/latest/ffmpeg-master-latest-win64-gpl.zip \
  -o /tmp/ffmpeg-win.zip
unzip /tmp/ffmpeg-win.zip -d /tmp/ffmpeg-win/
mkdir -p bin/win
cp /tmp/ffmpeg-win/ffmpeg-master-latest-win64-gpl/bin/ffmpeg.exe bin/win/ffmpeg.exe
```

### Verify encoders and drawtext

```bash
./bin/linux/ffmpeg -hide_banner -encoders | grep -E "h264_nvenc|h264_amf|h264_qsv|libx264"
./bin/linux/ffmpeg -hide_banner -filters  | grep drawtext   # must print a line, else text overlays won't render
```

## Docker Build Environment

A `Dockerfile` is included for building the app from any Linux host without polluting your local environment. The image bundles:

- **Node.js 24** (slim base)
- **Wine + Mono** вЂ” required for cross-compiling the Windows portable exe from Linux
- **Electron binary pre-cached** вЂ” downloads and caches the Electron zip at image-build time so `npm install` inside the container does not re-download it on every run

### 1 вЂ” Build the image

Run this once from the repository root. The image tag must match the one used in the run commands below.

```bash
docker build -t electron-app:builtin-packages .
```

> The build step downloads the Electron binary (~100 MB) and installs npm dependencies into the image layer. Subsequent runs reuse the cache and are fast.

### 2 вЂ” Run a build

The project directory is mounted as a volume so build output lands directly in `dist-electron/` on the host. FFmpeg binaries must already be present in `bin/` before running (see [FFmpeg Setup](#ffmpeg-setup)).

> **Cache the electron-builder toolchain.** The `-v moments-eb-cache:/eb-cache -e ELECTRON_BUILDER_CACHE=/eb-cache` flags below persist electron-builder's downloaded tools (winCodeSign, NSIS, app-builder) in a named Docker volume. Without them, `--rm` discards the cache and these are re-downloaded on every run. See [Build caching](#build-caching) below.

**Linux AppImage:**
```bash
docker run --rm -it \
  -v $(pwd):/app \
  -v moments-eb-cache:/eb-cache -e ELECTRON_BUILDER_CACHE=/eb-cache \
  -w /app \
  electron-app:builtin-packages /bin/bash -c "npm install && npm run build:linux"
```

**Windows portable exe** (cross-compiled via Wine):
```bash
docker run --rm -it \
  -v $(pwd):/app \
  -v moments-eb-cache:/eb-cache -e ELECTRON_BUILDER_CACHE=/eb-cache \
  -w /app \
  electron-app:builtin-packages /bin/bash -c "npm install && npm run build:win"
```

**Both targets in one pass:**
```bash
docker run --rm -it \
  -v $(pwd):/app \
  -v moments-eb-cache:/eb-cache -e ELECTRON_BUILDER_CACHE=/eb-cache \
  -w /app \
  electron-app:builtin-packages /bin/bash -c "npm install && npm run build:all"
```

### Build caching

Because the container runs with `--rm`, anything written to its filesystem is discarded when it exits. electron-builder downloads its toolchain (winCodeSign, NSIS/nsis-resources вЂ” used even for the portable target вЂ” and app-builder) into `~/.cache/electron-builder` at build time, so without a persistent cache it re-downloads them every run.

The `-v moments-eb-cache:/eb-cache -e ELECTRON_BUILDER_CACHE=/eb-cache` flags fix this: a **named** Docker volume (`moments-eb-cache`, auto-created on first use) holds the cache between runs, and `ELECTRON_BUILDER_CACHE` points electron-builder at it. The Electron binary itself is already baked into the image (`~/.cache/electron`), so it isn't affected.

- Only the **first** build downloads the toolchain; later builds reuse it.
- To inspect or reset the cache: `docker volume inspect moments-eb-cache` / `docker volume rm moments-eb-cache`.
- Don't replace the named volume with a bind mount on `~/.cache` (e.g. `-v $(pwd)/.cache:/root/.cache`) вЂ” a bind mount starts empty and would shadow the image's pre-cached Electron binary, forcing *that* to re-download too.
- Optional: add `-v moments-npm:/root/.npm` to also cache npm's download metadata across runs.

### Notes

- `npm install` is run inside the container on each invocation because native addons must be compiled for the container's glibc, not the host's. The installed `node_modules` are written into the mounted volume and will appear on the host вЂ” this is expected.
- Build output is written to `dist-electron/` in the mounted volume, so the AppImage / exe are immediately accessible on the host after the container exits.
- The `--rm` flag removes the container after each run. No state is kept between runs except what is written back through the volume mount (and the named cache volume above).

## GPU Acceleration

### How it works

When the Export dialog opens, the app runs a quick smoke-test (1-frame **256Г—256** encode) for each HW encoder found in the bundled FFmpeg binary. The best passing encoder is selected automatically and shown in the Export dialog. (The probe is 256Г—256 because Ampere/Ada NVENC rejects frames below its minimum dimension вЂ” a smaller probe would falsely fail on working GPUs. A failed probe logs `[GPU] smoke test failed вЂ¦` with the FFmpeg reason on the main-process stdout.)

Users can override the encoder selection before starting export. If a GPU encoder causes issues (driver bugs, codec negotiation failures at specific resolutions), switching to CPU and retrying is always available without closing the dialog.

### Encoder priority

| Priority | Encoder | Hardware | Speed vs CPU |
|---|---|---|---|
| 1 | `h264_nvenc` | NVIDIA GPU | 5вЂ“15Г— faster |
| 2 | `h264_amf` | AMD GPU | 4вЂ“10Г— faster |
| 3 | `h264_qsv` | Intel iGPU | 2вЂ“5Г— faster |
| 4 | `h264_v4l2m2m` | Linux V4L2 | 1.5вЂ“3Г— faster |
| 5 | `libx264` | CPU | baseline |

### GPU driver requirements (user's machine)

| Encoder | Requirement |
|---|---|
| NVENC | NVIDIA driver 522+ |
| AMF | AMD Adrenalin driver (recent) |
| QSV | Intel Media Driver вЂ” Linux: `intel-media-va-driver-non-free` |

The app detects and falls back gracefully if drivers are absent.

### Export flow

```
1. User opens Export dialog
   в†’ GPU detection runs (smoke-tests each HW encoder)
   в†’ GPUSelector shows available encoders

2. User clicks Start Export
   в†’ Renderer writes source files to OS temp dir
   в†’ Renderer builds FFmpeg step list (args with __ENCODER__ tokens)
   в†’ api.startExport({ steps, encoderOverride }) в†’ main process

3. Main process executes steps
   в†’ Resolves __ENCODER__ / __ENC_ARGS__ with detected encoder
   в†’ Spawns: ffmpeg -hide_banner [resolved args]
   в†’ Streams log lines back to renderer in real time
   в†’ Steps with fallbackOnFail (e.g. audio-less clips) recover silently

4. After all steps complete
   в†’ Renderer reads output.mp4 from temp dir
   в†’ Native save dialog в†’ user picks destination
   в†’ File copied, temp dir cleaned up
```

## Troubleshooting

| Problem | Fix |
|---|---|
| Export fails with GPU encoder | Open Export dialog, switch to CPU, retry |
| "FFmpeg not found" on launch | Run `node scripts/download-ffmpeg.js` or place binary in `bin/` |
| NVENC not detected (NVIDIA GPU present) | Update to NVIDIA driver 522+. Launch from a terminal and check for `[GPU] smoke test failed вЂ¦` вЂ” it prints the FFmpeg reason. Common causes: missing encode runtime lib (`libnvidia-encode`; note CUDA/AI workloads use a *different* lib), driver too old, or a datacenter GPU with no NVENC block. Over RDP the app auto-retries with `-hwaccel cuda`; otherwise use the GPU Selector to switch to CPU |
| NVENC fails over Windows RDP | App retries smoke-test with `-hwaccel cuda` automatically. If still failing, switch to CPU in the Export modal GPU Selector |
| QSV not detected (Intel iGPU present) | Install `intel-media-va-driver-non-free` (Linux) |
| Windows build fails (NSIS error) | Target is `portable` by default for Linux cross-compile вЂ” this is correct. For a full installer, build on a native Windows machine using the `_win_nsis` config in `package.json` |
| Wine not found for Windows cross-build | `apt-get install -y wine wine64 libwine mono-runtime` |
| Text overlay export fails on Windows | Font paths with drive letters (e.g. `C:/...`) are escaped as `C\:/...` in the drawtext filter вЂ” this is correct. Do not remove the escaping |
| White screen on launch | Check `electron/main.js` вЂ” verify Vite URL or `dist/index.html` path |
