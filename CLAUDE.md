## Project Overview

**Moments** is a cross-platform desktop photo & video slideshow editor built with Electron. Users import media, arrange clips on a timeline, apply transitions and text overlays, and export a finished MP4. Video encoding runs via a **native FFmpeg child process** with GPU acceleration ‚Äî replacing the original browser-based WebAssembly engine entirely.

**Platforms:** Linux (x64) ¬∑ Windows (x64)

**Stack**

| Layer | Tool |
|---|---|
| Shell | Electron 35 |
| Framework | React 19 + Vite 8 |
| Video encoding | Native FFmpeg binary (bundled) via IPC child process |
| GPU acceleration | NVENC (NVIDIA) ‚Üí AMF (AMD) ‚Üí QSV (Intel iGPU) ‚Üí CPU |
| Icons | `lucide-react` |
| Styling | CSS Modules |
| State | Custom hook (`useMediaStore`) ‚Äî no external state lib |

**Run locally**
```bash
npm install
# Place FFmpeg binary at bin/linux/ffmpeg (Linux) or bin/win/ffmpeg.exe (Windows)
npm run dev   # starts Vite on :5173 + Electron
```

## ‚öÝÔ∏è Export Pipeline ‚Äî DO NOT BREAK

**The export pipeline is extremely sensitive to changes.** Any modification to `useFFmpeg.js` must preserve all of the following invariants or the export will produce corrupt output or fail silently.

### Threading

Native FFmpeg has no threading restrictions. GPU encoders manage their own parallelism internally. For the CPU fallback (`libx264`), `-threads 0` is used to allow FFmpeg to use all available cores.

### Temp directory lifecycle ‚Äî CRITICAL

Source files, intermediate segments, and the final output all live in an OS temp directory created at the start of each export (`os.tmpdir()/moments_export_XXXXX`).

**The main process (`electron/main.js`) must NOT delete this directory in its `cleanup()` function.** The renderer reads the output file via `api.readFile()` after `startExport()` resolves ‚Äî deleting the temp dir in main's cleanup races with that read and causes ENOENT.

The renderer's `finally` block owns cleanup: it reads the output, copies it to the user's chosen save path, then calls `api.rmdir(tmpDir)`.

```
Renderer finally block order (MUST NOT change):
  1. api.readFile(outputFile)     ‚Üê read output from temp dir
  2. api.saveFileDialog()         ‚Üê user picks save location
  3. api.copyFile(tmp, savePath)  ‚Üê copy to final destination
  4. cleanup() ‚Üí api.rmdir(tmpDir) ‚Üê NOW safe to delete
```

### Audio stream fallback ‚Äî CRITICAL

Video clips without an audio stream must not cause the export to fail. The audio extraction step uses `-map 0:a:0` (explicit first audio stream). If no audio stream exists, FFmpeg exits non-zero and the main process automatically runs the `fallbackOnFail` silence step.

### Encoder tokens

The renderer builds FFmpeg argument arrays with two special tokens that main.js resolves at execution time:

- `__ENCODER__` ‚Üí replaced with the selected HW encoder string (e.g. `h264_nvenc`, `libx264`)
- `__ENC_ARGS__` ‚Üí spread-replaced with the encoder's quality arg array

### `IMAGE_EXTS` must be declared inside `exportMoment()`

```js
// Required at the top of exportMoment() try block:
const IMAGE_EXTS = new Set(['jpg','jpeg','png','gif','webp','heic','avif'])
```

### Font path escaping on Windows ‚Äî CRITICAL

FFmpeg's `drawtext` filter uses `:` as its option separator. Windows drive letter colons (e.g. `C:/path`) must be escaped as `C\:/path` ‚Äî **single quotes do NOT protect colons** in FFmpeg filter syntax. This is handled in `buildDrawtext`:

```js
const escapedPath = fontPath
  .replace(/\\/g, '/')            // backslash ‚Üí forward slash
  .replace(/^([A-Za-z]):/, '$1\\:') // C: ‚Üí C\:
  .replace(/'/g, "\\'")           // escape single quotes
```

Do not remove or simplify this escaping ‚Äî it will silently break text overlay export on Windows.

### Source file index consistency ‚Äî CRITICAL

When building `validClips`, each clip's associated temp files (`src_N`, `aud_N`, `seg_N`) must all use the same index `N` ‚Äî the clip's position in the `validClips` array, **not** its position in the original `clips` array. Clips with missing files are skipped, so using the original index creates a mismatch.

```js
const vi    = validClips.length   // index BEFORE push
const fname = `src_${vi}.${ext}`
validClips.push({ ...c, _fname: fname, ... })
```

### Font handling ‚Äî native paths

Fonts are referenced by real OS paths resolved via `api.fontPath(fontFile)`. There is no WASM FS write step. Each **distinct** custom font (from `seg.customFontData`) is written to the temp dir once as `font_<family>.ttf` (keyed by `customFontFamily`) before the step list is sent to main ‚Äî so captions using different uploaded fonts each get the right face, and a font reused across captions is written once.

Custom font bytes must still be `.slice()`'d when stored into React state to avoid detached ArrayBuffer errors on repeated exports.

### Custom fonts ‚Äî stable family, reuse, persistence

On upload, the bytes get a **content-stable family** `customFontFamily = MomCustom_<hash>` (`hashFontBytes` in `src/textLayout.js`). This drives everything:
- **Preview** (`Preview.jsx`): registers **every** caption's custom font once into a module-level `_loadedCustomFonts` cache (a `FontFace` per family, never removed), and `bumpFonts` forces a repaint when a face finishes loading. So an upload shows immediately, a re-upload swaps cleanly (new family), multiple captions can use different custom fonts, and a font is **never dropped** when the caption is deselected or its bytes detach post-export. `famFor` returns the segment's family.
- **Export**: per-family temp files (above); the measure family is `seg.customFontFamily`.
- **Workflow**: `saveWorkflow` already stores custom fonts as base64 in a deduped `customFonts` table (keyed by a name+content hash) and preserves `customFontFamily`; `loadWorkflow` restores the bytes and derives `customFontFamily` from them for older workflows. So the same font reused on other captions persists once and re-links on load.

## ‚öÝÔ∏è GPU Acceleration

### Detection

On each export open, the main process runs `ffmpeg -encoders` to check compiled-in HW encoders, then smoke-tests each with a **1-frame 256√ó256 encode**. Results are cached per session.

> ‚öÝÔ∏è **Probe size matters.** The smoke frame must be **256√ó256** ‚Äî Ampere/Ada NVENC rejects frames below its minimum dimension (`InitializeEncoder failed: ‚Ä¶ Frame Dimension less than the minimum supported value`), so a smaller probe (the old 64√ó64) falsely fails on perfectly working GPUs and silently drops to CPU. Real exports are ‚â•854√ó480, so they were never affected. The probe captures stderr and logs `[GPU] smoke test failed ‚Ä¶` (last few FFmpeg lines) on any non-zero exit, so a real failure surfaces its reason instead of vanishing into "no GPU". The log goes to the **main-process stdout** (visible when launched from a terminal), not the in-app Log drawer.

### Priority

| Priority | Encoder | Hardware |
|---|---|---|
| 1 | `h264_nvenc` | NVIDIA GPU |
| 2 | `h264_amf` | AMD GPU |
| 3 | `h264_qsv` | Intel iGPU |
| 4 | `h264_v4l2m2m` | Linux V4L2 |
| 5 | `libx264` | CPU (always available) |

### Encoder quality args ‚Äî `encoderQualityArgs(encoder, tier, bitrateMbps)`

`electron/main.js`. The **Export modal** exposes three quality tiers; each maps to
a constant-quality target (CRF/CQ) + a speed‚Üîefficiency preset, H.264 **high**
profile + `yuv420p` for universal playback. A **custom bitrate** (Mbps), when set,
overrides constant-quality with capped VBR (ABR + `-maxrate`/`-bufsize`). The tier
+ bitrate ride in the `ffmpeg:export` IPC payload (`exportQuality`, `exportBitrate`)
and are persisted as global prefs (`moments.exportQuality` / `moments.exportBitrate`),
not in the workflow.

| Tier | CRF/CQ | libx264 preset | NVENC preset |
|---|---|---|---|
| `high` | 18 | slow | p6 |
| `balanced` (default) | 21 | medium | p4 |
| `small` | 26 | veryfast | p2 |

> Presets are slower than the old `ultrafast`/`p2` (esp. CPU `high`/`balanced`).
> The GPU encoder smoke-test in `main.js` uses minimal args, not these ‚Äî so if a
> HW path passes detection but rejects a fuller arg here, the encode step fails
> (no automatic CPU fallback for the main encode). NVENC/libx264 are the tested paths.

### Output formats (Stage 5)

`detectGPUCapabilities()` also reports codec availability (`vp9`/`opus`/`vorbis`/`gif`,
parsed from `ffmpeg -encoders`). The modal's format picker is gated on these; an
unavailable persisted format falls back to MP4 at export time. All formats are a
**final pass off the finished H.264 master** (`preWebFile`), so the fragile
per-clip/transition/text stages are untouched ‚Äî see Stage 5.

### Real progress + ETA

`exportMoment` tags each step with a `weight` (media-seconds it processes: clip
output length for Stage-1 encodes, the whole timeline for xfade/music/text/fade/
webm/gif passes, ~0 for stream-copies), sums them, then parses FFmpeg's streamed
`time=` per step (`onLog`) to drive an accurate bar + ETA ‚Äî replacing the old
counter that advanced as steps were merely *queued*. ETA surfaces in the modal.

```js
  animation:    string,       // 'fade' | 'slide' | 'typewriter' ‚Äî all produce fade in/out in export
  fontSize:     number,       // Preview px units (range 30‚Äì400, default 144); scaled to canvas width at export
  color:        string,       // Hex e.g. '#ffffff'
  fontFile:     string,       // Preset key or 'custom'
  customFontName: string | null,
  customFontData: Uint8Array | null,
  customFontFamily: string | null, // 'MomCustom_<hash>' ‚Äî content-stable @font-face name (see Custom fonts)
  position:     string,       // 'bottom' | 'top' | 'center' | 'custom'
  posX:         number,       // % (0‚Äì100), used when position === 'custom'
  posY:         number,       // % (0‚Äì100), used when position === 'custom'
  textAlign:    string,       // 'left' | 'center' | 'right' ‚Äî alignment of wrapped lines within the box
  boxWidth:     number,       // % of canvas width ‚Äî wrap width / max text box width (default 80)
}
```

### Text wrapping & alignment parity (`src/textLayout.js`)

The preview and export wrap captions identically because a **shared** `wrapText(text, fontFamily, fontSize, boxWidthPct)` (offscreen-canvas `measureText` at a fixed `WRAP_REF_W=1280` reference; scale-invariant) computes the line breaks **once** and both renderers honor them literally:
- **Export** (`useFFmpeg.js` Stage 3) writes the `\n`-joined wrapped lines to the textfile, and `buildDrawtext` adds `text_align=<left|center|right>` + `line_spacing` (one `drawtext` per caption). `x=(w-text_w)/2` centers the block; `text_align` aligns lines within it.
- **Preview** (`Preview.jsx`) renders the same `wrapText` lines (`white-space: pre`, `width: fit-content`, `max-width: boxWidth%`, `text-align`).
- Both measure with the **resolved render font** (`famFor`): a CJK caption is measured/rendered entirely in Noto in **both** preview and export (drawtext is one-font-per-caption), so the preview previews the real export. Manual `\n` are honored as hard breaks; over-long single words are hard-broken by character.

**Important:** `endTime` is never stored on text segments. `buildDrawtext` always derives it as `startTime + duration`. Never add an `endTime` field to the segment schema.

## Export Pipeline ‚Äî `useFFmpeg.js` ‚Üí `exportMoment()`

### Stage 1 ‚Äî Per-clip encoding ‚Üí `seg_N.mp4`

Every `seg_N.mp4` delivered to Stage 2 must conform exactly:

| Property | Value |
|---|---|
| Video codec | `__ENCODER__` (GPU or libx264) |
| Pixel format | yuv420p |
| Frame rate | 30 fps CFR |
| Timebase | 1/90000 |
| Audio codec | AAC 128k stereo 44100 Hz |

#### Clip transform path (Phase A ‚Äî rotate / resize / move)

When a clip has a non-identity transform (`clipHasTransform(c)` ‚Üí `scale‚âÝ1 || rotation‚âÝ0 || offsetX‚âÝ0 || offsetY‚âÝ0`), the **non-blur** image and video paths route through `buildTransformOverlay(srcLabel, bgLabel, W, H, c, extra)` instead of fit‚Üípad:

```
fit (decrease, NO pad) ‚Üí [eq/speed] ‚Üí format=rgba ‚Üí scale=iw*S:ih*S
  ‚Üí rotate=Œ∏rad:fillcolor=none (ow/oh grown to the rotated bbox, transparent corners)
  ‚Üí overlay onto [bg] at (main_w-overlay_w)/2 + offsetX¬∑main_w : (main_h-overlay_h)/2 + offsetY¬∑main_h
```

- `bg` = `color=c=black:s=WxH` for non-blur clips, or the **blurred fill** (`[_bg]`) for blur-background clips ‚Äî the blur paths split the source, build `[_bg]`, then composite the transformed fg over it via `buildTransformOverlay('[_fg0]', '_bg', ‚Ä¶)`. The overlay clips to the canvas, matching the preview's `overflow:hidden`.
- **Geometry is verified against the preview** via an offscreen-render calibration harness (headless Chromium replica of the preview CSS vs FFmpeg, pixel-compared). FFmpeg `rotate=+rad` matches CSS `rotate(+deg)` (both clockwise); uniform scale commutes with rotation so the FFmpeg scale-then-rotate equals the CSS rotate-then-scale.
- **Mutually exclusive** with Ken Burns (`imageEffect`) ‚Äî a transform sets `effectVf = null` (transform wins). In the **preview**, the transform is applied to the foreground media element only; the blur background is a separate full-canvas element, so it's unaffected ‚Äî matching the export.
- Preview mirror: `Preview.jsx` applies `transform: translate(offsetX%, offsetY%) scale(scale) rotate(rotation deg)` (translate % is % of the canvas since the media element is 100% of the canvas-aspect screen).

#### Image clip path

Single FFmpeg invocation using `filter_complex`: fit ‚Üí even-align ‚Üí pad ‚Üí optional effect/EQ.

#### Image clip ‚Äî blur background path

```
[0:v] split=2 ‚Üí [_bg0][_fg0]
[_bg0] scale W:H (fill) ‚Üí crop W:H ‚Üí boxblur ‚Üí [_bg]
[_fg0] scale W:H (fit, letterboxed, NO padding) ‚Üí even-align ‚Üí [_feven]
       optional eq ‚Üí [_feven or _fpadded]
[_bg][_feven] overlay=(main_w-overlay_w)/2:(main_h-overlay_h)/2 ‚Üí [vout]
```

- The fg is **never padded to W√óH** after fitting. Padding makes `overlay_w = main_w` ‚Üí offset collapses to `0:0` ‚Üí blurred bg is hidden entirely.
- The fg stays at its natural letterboxed size so the blurred bg shows through the bars.
- `overlay` uses FFmpeg runtime vars (`main_w`, `main_h`, `overlay_w`, `overlay_h`) ‚Äî not JS template vars.

#### Video clip path

Two steps: audio extraction (`-map 0:a:0` with silence fallback), then placement encode (fit ‚Üí pad ‚Üí speed/eq).

#### Video clip ‚Äî blur background path

Same filter_complex strategy as image blur. Speed and eq are applied in the fg chain after the fit step.

### Stage 2 ‚Äî Transitions, music ‚Üí `prefinal.mp4`

Three paths:
- **Path A** ‚Äî concat demuxer stream copy (all cuts, no music)
- **Path B** ‚Äî single clip copy or music mix
- **Path C** ‚Äî xfade/acrossfade filter_complex (mixed transitions or music)

### Stage 3 ‚Äî Text overlays ‚Üí `output.mp4`

All `drawtext` filters in one `-vf` pass. Font files referenced by real OS paths. Custom fonts written to temp dir before step list is sent. Caption **text** is never inlined into the filter string ‚Äî each caption is written to `text_N.txt` and referenced via `textfile=` with `expansion=none`, so special characters can't break the chain (see Bug History). Windows drive letter colons escaped as `C\:/` ‚Äî see Font path escaping section above.

Text timing uses `enable='between(t,startTime,startTime+duration)'`. The alpha expression drives fade animation for `animation === 'fade'` (and `'slide'`/`'typewriter'` as fallback ‚Äî FFmpeg drawtext has no native slide or typewriter effect).

### Stage 4 ‚Äî End fade ‚Üí `output_final.mp4` (optional)

Applied only when `endFadeVideo` or `endFadeAudio` is true.

### Stage 5 ‚Äî Deliverable format

A final pass off the finished H.264 master (`output.mp4` / `output_final.mp4`),
selected by `exportFormat`:

- **MP4** (`output_web.mp4`) ‚Äî stream-copy with `-movflags +faststart` (moov atom
  to the front ‚Üí progressive online playback / faster uploads). Lossless, no
  re-encode.
- **WebM** (`output.webm`) ‚Äî `libvpx-vp9` (CRF by tier, or custom bitrate) +
  `libopus`/`libvorbis` audio (`-an` if neither available). Slower (VP9).
- **GIF** (`output.gif`) ‚Äî two-pass `palettegen`/`paletteuse` (lanczos, capped fps
  15 / width 640). Silent, looping.

The renderer reads/saves whichever `outputFile` the branch produced, with a
matching blob MIME, file extension, and save-dialog filter. (A double-encode for
WebM/GIF ‚Äî H.264 master ‚Üí VP9/GIF ‚Äî is the deliberate trade for keeping the
sensitive pipeline untouched.)

## Preview music playback ‚Äî `Preview.jsx`

## Known Gaps / Future Work

- **`fileDuration` for direct timeline drops** ‚Äî when files are dragged directly to the timeline (bypassing the media panel), `fileDuration` is `null`. The TrimBar does not appear until the user previews the clip in the media panel first.
- **Speed + trim interaction** ‚Äî `actualDur = (effectiveTrimEnd - trimStart) / speed`. Correct for wall-clock output duration.
- **Music file not embedded in workflow** ‚Äî the audio bytes aren't stored, but the **filename is saved and re-linkable** (same name-match approach as clip media): on load, `musicFileName` is remembered and the Media panel shows a "Re-link music" prompt; re-adding the matching file restores playback while **keeping** the saved `musicTrimStart`, `musicTrimEnd`, and `musicVolume`. Re-saving without re-adding preserves the pending link.
- **No undo/redo** ‚Äî all mutations are direct `setState` calls.
- **Blur background preview is a CSS approximation** ‚Äî `Preview.jsx` shows a `blur()/scale` cover copy behind the letterboxed foreground (`.blurBg`); it approximates, but won't exactly match, FFmpeg's `boxblur` fill.
- **Per-clip transition duration is applied in export** ‚Äî `exportMoment` resolves `tdFor(clip) = clip.transitionDuration ?? global td` per boundary (0 for `none`); every segment is padded to `maxTd + 0.5` so xfades always fit. End-fade total and the store's `exportDuration`/`timeline` all use the same per-clip math.

### Settings persistence
- `aspectRatio` and `quality` persist across sessions via `localStorage` (`moments.aspectRatio` / `moments.quality`), seeded into the store's initial state (no flash, no undo entry). The wrapped `setAspectRatio`/`setQuality` write back on every change ‚Äî including workflow loads and undo (last-used wins).
- **Multiple custom fonts** ‚Äî supported: each distinct upload gets a content-stable `customFontFamily` and its own `font_<family>.ttf` at export; the preview registers each as a separate `@font-face`. (Previously limited to one shared `font_custom.ttf`.)
- **Media re-matching is filename-only** ‚Äî on workflow load, clips are matched by filename alone.
- **`slide` and `typewriter` text animations** ‚Äî these are CSS-only effects in the Preview. Both fall back to a fade in/out in the FFmpeg export, since `drawtext` has no native slide or typewriter support. A future approach would be to render text as an overlay video using `lavfi` or a separate compositing pass.
- **Brightness preview vs export discrepancy** ‚Äî CSS `filter: brightness()` is multiplicative while FFmpeg `eq=brightness=` is an additive luma offset. The visual result differs at high values. A future fix would switch the export to `curves` or `lutyuv` for a true multiplicative match.
- **Contrast/saturation scale** ‚Äî CSS uses `/100` divisor, FFmpeg uses `/50`. Export effect is approximately 2√ó stronger than the preview at any given slider value.

## Bug History (resolved)

### Text overlay timing
`buildDrawtext` was reading `seg.endTime` ‚Äî a field never set anywhere. Fixed to derive `endTime = startTime + duration`. Text overlays now respect their configured duration.

### Text overlay animation
`alphaExpr` was gated on `seg.fadeIn` ‚Äî a field never set anywhere, so all overlays exported with flat `alpha='1'`. Fixed to drive the fade expression from `seg.animation === 'fade'`.

### Source file index mismatch after workflow load
`validClips` was naming temp files using the original `clips` array index. When clips with missing files were skipped (common after workflow load), the `src_N`, `aud_N`, `seg_N` indices diverged, causing audio/video step mismatches. Fixed to use `validClips.length` before push as the consistent index.

### Blur background ‚Äî fg covering blur entirely
Two bugs worked together: (1) the overlay expression used JS template vars `(W-w)/2` where `w` resolved to the overlay's own width, collapsing the offset to zero; (2) the fg was padded to `W√óH` after fitting, making it the same size as the bg. Both fixed: overlay now uses `(main_w-overlay_w)/2:(main_h-overlay_h)/2`, and the pad step is removed so the fg stays at letterboxed size.

### Windows text overlay export failure
FFmpeg's `drawtext` filter uses `:` as its option separator. Windows drive letter colons (`C:/path`) were not escaped, causing the filter parser to split the path at the colon. Fixed by escaping `C:` ‚Üí `C\:` in `buildDrawtext`.

### Text overlays silently dropped on captions with special characters
Caption text was inlined as `drawtext=‚Ä¶:text='<escaped>'`. The escape pass ran in the wrong order (`'` ‚Üí `\'` *before* `\` ‚Üí `\\`), so an apostrophe became `It\\'s`, whose `'` closed the quote early and produced an invalid `-vf` chain. Because **all** captions share one `-vf` chain, one bad caption made the whole text pass fail ‚Üí the `s3-text` `fallbackOnFail` copied the video through with **zero overlays**. Reported as "text doesn't show, especially with trimmed clips / a mix of fonts" ‚Äî really any caption containing `'`, `:`, `,`, `[`, `]`, `%`, or `\`. Fixed by **not inlining text at all**: each caption is written to `text_N.txt` in the temp dir and referenced via drawtext `textfile=` (only the path is escaped, via the same `escPath` used for fonts), plus `expansion=none` so drawtext does no `%{...}` expansion on the file content (a bare `%` like "100%" otherwise throws "Stray %"). Verified with a multi-font, multi-caption smoke test covering all the above characters.

### CJK (Korean/Chinese/Japanese) text overlays missing from export
Captions in Korean/Chinese rendered fine in the preview but exported as blank/tofu ‚Äî only Latin text showed. Cause: every bundled preset font is Latin-only, and FFmpeg `drawtext` uses one font face per caption with no per-glyph fallback, so it draws `.notdef` boxes (or nothing) for CJK codepoints. The preview "worked" only because the browser does CSS per-glyph fallback to OS fonts. ### Windows build icon error

### Windows build NSIS uninstaller error
`win.target` was set to `nsis`. NSIS uninstaller stub generation fails when cross-compiling from Linux. Fixed by switching the default target to `portable`. The NSIS config is preserved as `_win_nsis` for native Windows builds.
